一、Mirror Reserves（镜像储备金）↗ 原文

核心相关角色🧑‍🌾 双边 LP ⭐⭐⭐📈 杠杆交易者 ⭐⭐💰 单边出借者 ⭐⭐

直觉理解：金库的"欠条账本"

🏦

real reserves

合约实际持有的 token 数量
——金库里的黄金

📋

mirror reserves

已借出但未归还的虚拟记账值
——金库开出的黄金欠条

💰

lend reserves

单边出借者存入的资金
——存款人的账户余额

为什么需要 Mirror Reserves？ 在不铸造 ERC-20 合成 token 的前提下，通过在账本上增减 mirror 值，维持恒等式成立，从而精确追踪所有杠杆/借贷头寸的净敞口，同时节省 gas，保持系统自包含。

扩展版 AMM 恒等式

( x + x' ) × ( y + y' ) = k
x real ETH 储备 x' mirror ETH（欠条） y real USDT 储备 y' mirror USDT（欠条）

与传统 AMM Real Reserves 的本质区别

特征Real Reserves（Uniswap V2 风格）Mirror Reserves（Likwid）
本质池中实际存在的可交易 token账本记载的已借出、未归还的虚拟数值
物理存在有，是合约地址持有的真实余额无，是协议内部状态变量中的数字
经济含义代表池的即时兑付能力（流动性深度）代表协议对 LP 的债务，以及杠杆交易者的风险敞口
变动驱动用户交易、添加/移除流动性借贷发生/偿还、杠杆仓位开立/关闭（含清算）
可提取性LP 可按份额随时赎回不可直接赎回，债务偿还时自动减少

Alice 看空 ETH，存入 100 ETH 抵押，开 2x 空头（借出同等 ETH 卖出，押注 ETH 下跌后低价买回还债获利）。

每做空不同 token 都需要单独保证金吗？

是的。Likwid 是隔离保证金（Isolated Margin）模型——每个 Vault 对应一个交易对（ETH/USDT、SOL/USDT 分别是独立 Vault），仓位以 NFT 绑定在特定 Vault 内。做空 ETH 需要将抵押品存入 ETH Vault，做空 SOL 需要存入 SOL Vault，两者不共用保证金。优势：单个 Vault 的风险不传染其他 Vault；代价：资本效率低于 CEX 的统一保证金账户。

价格影响如何波及所有用户 步骤3 完成后，该 Vault 内 ETH 价格从 1.000 → 0.764（跌幅 23.6%）。由于所有仓位的盈亏、margin level、清算价都基于同一个 Vault 的池内价格计算：
  • 其他 LP 提供者：持仓价值随之重新定价，实际承受无偿损失（Impermanent Loss）
  • 其他多头持有者：ETH 价格下跌 → 其 margin level 也同步恶化 → 可能触发级联清算
  • 其他空头持有者：ETH 价格下跌 → margin level 改善，获益
这是 Likwid 共享池模型的核心风险：每一笔大额杠杆交易都在「移动」池内定价，直接影响同 Vault 内的所有其他参与者。
平仓（正常还款，ETH 如期下跌） ETH 价格继续下跌后，Alice 用 83.3 USDT 低价买回 >100 ETH → 还清 100 ETH 债务 → mirror_x 归零 → 抵押品归还，价差即为盈利。欠条销毁，账本平衡。

三、清算机制：liquidateBurn 与 liquidateCall ↗ 原文

核心相关角色⚡ 清算人 ⭐⭐⭐📈 杠杆交易者 ⭐⭐⭐🧑‍🌾 双边 LP ⭐⭐
什么是清算？ 在杠杆借贷系统中，若资产价值下跌或债务价值上涨，抵押品可能不再足以覆盖债务。清算是在仓位变成系统净负债之前强制关闭它——用剩余抵押品偿还债务，阻止坏账蔓延到其他流动性提供者。

为什么 Likwid 需要自己的清算机制？

为什么 Likwid 必须自己处理清算，而不能像 Aave 那样外包？

Aave 的做法：借款时，协议给借款人发一个可以在市场上自由买卖的「债务凭证」代币。任何人都能买这个凭证来参与清算，市场上形成了竞争激烈的清算机器人生态——就像公开拍卖。

Likwid 的做法：借款的「欠账」只是 Vault 内部账本上的一个数字（mirror_x），像私人记账本里的一条备注，既不是代币，也无法转让。没有公开市场，外部套利者无法低门槛地参与竞价清算。

所以 Likwid 设计了自己的两条路径：优先在 Vault 内部完成清算（清算人只需要触发操作，不需要带钱来）；内部解决不了时，才让外部人带钱进来偿债。

🧭 两种 LP 模式，而非"一份钱三种用途"

Likwid 的 Vault 有两种独立的 LP 模式，对应不同的收益结构和风险：

① 双边 LP（Bilateral LP）
存入 ETH+USDT 配对资金，进入 pair reserves，充当：
  • swap 对手方（AMM 流动性深度）
  • 间接支撑杠杆可行性（pair+lend 共同构成 vault 总深度）
收益：swap 费（含杠杆开平仓产生的 swap 费）
风险：IL + 池深度不足时的流动性风险
⚠️ 借贷利息流入 lend reserves，由单边出借者专属获得，不流向 pair LP
② 单边出借（Lend Reserves）
只存单一币种，不做市，纯作为杠杆方借贷来源。
  • 不进入 AMM 曲线
  • 不承担 IL
收益：借贷利息
风险：利用率波动 + 极端坏账（保险基金兜底）

核心：资本效率高的本质是 pair reserves（双边 LP）和 lend reserves（单边出借）共存于同一个 Vault，两者共同构成 swap 深度和杠杆借贷的资金池——而非像 Uniswap + Aave 拼接那样资金分散在两个独立协议。

💡 双边 LP = swap 做市，earns swap fees；单边 LP = 专门提供借贷资金，earns 借贷利息。两种模式收益来源不同，但共享同一个 Vault 深度。

两条路径的定位

🔥

liquidateBurn — 首选路径

Vault 内部执行 AMM swap，用抵押品买回债务资产，Mirror Reserve 归零。清算人只需 Gas，无需垫资。适用于池深充足场景。

📥

liquidateCall — 备用路径

池子过浅、内部 swap 滑点过高时启用。清算人从外部注入债务资产直接偿债，无额外价格冲击，但需清算人自备资金。

命名注意（Likwid 专属术语） liquidateBurn / liquidateCall 是 Likwid 合约函数名，不是 DeFi 通用术语。Aave 叫 liquidationCall()，GMX 叫 liquidatePosition()——本质都是清算，但执行路径不同。

触发条件

Margin Level = 资产价值 / 债务 ≤ 阈值
资产价值：基于当前池状态（含储备感知定价）  |  债务：借款本金 + 累计利息
阈值不是固定 1.1——这是每个 Pool 初始化时设定的参数（称为 marginRatio）。
低波动池（如稳定币）：1.05；高波动池（如 meme coin）：1.15~1.2。1.1 是文档示例值。

场景：ETH 反涨触发空头清算（延迟清算 → 坏账示例）

⚠️ 为什么这个例子有坏账？ 正常情况下，清算应在 ML 降至阈值（1.10）时立即触发——此时 ETH 仅涨到 ≈0.757 USDT，资产仍超过债务 10%，liquidateBurn 的滑点通常在 10% buffer 内，不产生坏账。
此例展示的是延迟清算场景：无人及时调用合约，ETH 继续涨至 0.87 → ML 已跌破 1.0（资产 < 债务）→ 清算后必然出现坏账，由保险基金兜底。

liquidateCall 执行路径对比（外部还款）

1
清算人先 approve ETH 转入合约
清算人必须自备足额 ETH（此例 = 100 ETH）；调用前需 approve 合约地址
2
调用 liquidateCall(positionId, amount)，注入 ETH
ETH 直接注入合约，不经过 Vault 内部 swap，不产生额外价格冲击
3
合约用清算人的 ETH 直接偿还 mirror 债务
mirror_x 归零，债务消除；无坏账（清算人提供了足额资产）
4
清算人获得 Alice 的 USDT 抵押品（含清算折扣奖励）
清算折扣 = 补偿清算人的资金占用成本；Alice NFT 销毁
两者的核心区别 liquidateBurn：协议内部自动处理，清算人零资本（只需 Gas），但依赖池深度，可能留坏账。
liquidateCall：清算人亲自提供 ETH，不产生额外价格冲击，清算更干净，但需清算人持有足额 ETH。
Likwid 优先走 liquidateBurn，池子太浅（买回 ETH 的滑点过高）时 fallback 到 liquidateCall。
⚠️ 清算失灵场景：何时无人愿意清算？ liquidateCall 失灵：仓位已资不抵债（ML < 1.0）时，清算人提供 ETH 换到的 USDT < ETH 成本，净亏损 → 无经济激励，无人响应。本文例子（ETH=0.87，资产83.3 < 债务87）即是此情形。

liquidateBurn 相对更抗失灵：清算人只需 Gas，奖励从抵押品扣取，只要仓位够大奖励 > Gas 就有利可图。但仓位极小（尘埃仓位）时仍可能无人触发。

若真的无人清算 →
  1. 仓位继续累积负债，直到协议通过保险基金注销 NFT
  2. 保险基金耗尽 → 坏账直接稀释 LP（按份额承担损失）
  3. LP 亏损撤出 → 池子更浅 → 下次清算成本更高 → 清算激励进一步恶化 → 循环触发死亡螺旋
这是 Section 六「死亡螺旋」中价格失灵的另一个触发路径：不仅是价格暴跌造成，也可以由清算激励失效引发。

五、与 Aave / GMX 清算机制对比

核心相关角色⚡ 清算人 ⭐⭐⭐🧑‍🌾 双边 LP ⭐⭐📈 杠杆交易者 ⭐⭐

各协议清算机制展开说明

🏦 Aave — 谁的贷款出问题，谁来清算谁赚钱

Aave 是一个借贷平台：你把 ETH 存进去，可以借出 USDT（或反过来）。存的多、借的少才能借——叫「超额抵押」。

出问题时怎么处理？

  • 系统给每笔借款打一个「健康分」。分数 < 1，代表抵押品已不足以覆盖借款，任何人都可以来清算。
  • 来清算的人（通常是自动程序）帮你还钱，然后拿走你的抵押品，但能拿到比还的钱多一点（比如还了 100 USDT，拿走价值 108 USDT 的 ETH）。多出来的 8 USDT 是清算人的利润。
  • 清算人不需要自己有钱——他可以用「闪电贷」（一种在同一笔交易里借进来、用完即还的工具）零本金完成操作。
  • 风险是隔离的：每种资产有独立的市场，一个崩了不会传染其他资产。

对比 Likwid：Aave 的坏账被扔给了公开市场（清算人在 Uniswap 这类平台上卖掉抵押品）；Likwid 把坏账关在自己的池子里处理，不影响外面，但会影响同一个池子里的所有人。

📈 GMX v1 — 所有人的对手方是同一个「公共金库」

GMX 是做合约交易的（可以 1x、5x、甚至 50x 做多做空 BTC/ETH）。它没有订单簿，而是用一个叫 GLP 的大资金池当所有交易者的对手方。

类比：你在 GMX 做多 ETH，你赚的钱从 GLP 池里出；你亏的钱进入 GLP 池。GLP 持有者相当于「开了一家赌场的庄家」——交易者总体亏钱时，GLP 持有者赚；交易者整体赚钱时，GLP 持有者亏。

出问题时怎么处理？

  • GMX 自己运行监控程序（Keeper），当某个仓位的亏损达到保证金上限时，自动平仓。
  • 平仓不需要外部人来清算——Keeper 直接从 GLP 池扣钱处理，清算人只是触发操作拿点手续费。
  • 风险由 GLP 池全体持有者共同承担：某个大仓位爆仓，冲击被稀释在整个 GLP 里。

对比 Likwid：GMX 的 GLP 是全平台共享的大池子，风险被几百个市场稀释；Likwid 每个 Vault 各自独立，ETH/USDT 这个池子里的清算只能由这个池子自己消化。

⚡ dYdX v4 — 最像传统期货交易所的链上合约

dYdX 的体验最接近 OKX/Binance 合约：有挂单、有撮合，是真正的订单簿，不是 AMM。它运行在自己独立的区块链上（基于 Cosmos 技术），速度快、手续费低。

出问题时怎么处理？

  • 每个账户有「初始保证金率」和「维持保证金率」，类似 OKX 合约的强平机制。当亏损吃掉维持保证金时触发强平。
  • 强平时从订单簿上以市价成交，不像 AMM 有滑点问题——有人挂单就能成交。
  • 如果市价成交后仍有亏损（穿仓），由「保险基金」补足，资金来源是协议手续费的积累。
  • 每个子账户风险独立——一个账户爆仓不影响其他账户。

为什么 dYdX 不需要外部清算人来"抢"？

dYdX v4 的清算由区块链验证节点（Validator）在出块时自动扫描并触发。当某个账户保证金低于维持保证金率，协议自动在订单簿挂出市价强平单——不需要外部机器人发现并抢跑。这和 Aave 的"谁先发现谁赚奖励"竞争模式完全不同：dYdX 是协议内置自动化，Aave 是开放的清算人市场。
⚠️ 也不是"不需要清算"——而是清算执行方从"外部 MEV 机器人"变成了"协议出块逻辑自动执行"。

对比 Likwid：dYdX 的清算价格由市场买卖双方决定（谁挂单谁成交），更市场化；Likwid 的清算价格由池子内部的数学公式决定，极端行情下可能严重偏离真实市价。

🔷 Uniswap v3 — 纯交易所，不借钱、不杠杆、没有清算（对比参考）

Uniswap v3 就是一个自动做市商（AMM）：你把两种代币存进去，其他人来换，你赚手续费。没有借贷，没有杠杆，所以也没有清算。

存钱进去有什么风险？

  • 不是「被强平」的风险，而是「无常损失」（Impermanent Loss）：当你存入 ETH+USDT 后，如果 ETH 价格大涨，你池子里的 ETH 会被别人换走，你剩下的 USDT 多了、ETH 少了——相比你直接拿着 ETH，会少赚一些。价格波动越大，这个损失越明显。
  • 只要价格没跑出你设定的区间，你就持续赚手续费；一旦跑出去，你的钱变成全是一边（要么全是 ETH，要么全是 USDT），停止赚费用。

为什么列在这里：Likwid 的基础架构和 Uniswap v3 很像（都是集中流动性 AMM），区别在于 Likwid 在上面叠加了借贷层。这让 Likwid 的 LP 除了无常损失风险之外，还多了一层借款人违约的风险。

七、动态费率机制深度解析↗ 原文

核心相关角色🤖 套利者 ⭐⭐⭐🔄 现货交易者 ⭐⭐⭐🧑‍🌾 双边 LP ⭐⭐
研究背景 白皮书专设章节"Dynamic Fee Strategy Against MEV and Arbitrage Attacks"。Likwid 是为数不多在协议层将动态费率作为主动风险管理工具（而非收益分配工具）的 DeFi 协议。

核心机制原理

Likwid 动态费率的设计目标是：让套利成本 > 套利利润，从而保护 LP 并将套利价值的一部分转化为协议收入。

fee = f( ΔP, U )
ΔP 池内价格偏离外部参考的幅度 U 资金利用率（utilization） fee 实时调整的 swap 手续费

触发信号与调整逻辑

1
监控价格偏离（Bounded Price Movement）
持续比较 live reserves 定价 vs Truncated Oracle（截断预言机/TWAP）。当偏差超过预设阈值时，判定存在套利机会
2
监控资金利用率
高利用率 = 借贷需求旺盛 = 潜在闪电贷攻击风险上升 → 费率随利用率非线性增长，补偿 LP 风险
3
费率自动上调
超出阈值后费率自动提升，增加套利者成本。价格恢复稳定后费率回落至基础水平。Margin Fees 也受同一逻辑约束
4
与清算系统协同
动态费率是死亡螺旋三重防护的第一道防线（见第六节），在价格急剧偏离时主动减缓套利速度，为清算留出处置时间

与主流 DEX 费率机制对比

协议费率类型设计目标与 Likwid 的本质区别
Uniswap v3 / Pancake v3 固定（池创建时确定，不可变） LP 收益分配，简单可预测 静态 费率不响应市场状态，套利利润全归套利者
Raydium CLMM 固定，继承自池初始化设置 集中流动性资本效率 静态 "动态"仅指流动性分布，不涉及手续费调整
Trader Joe v2.1 离散 bin 价差（结构性静态） 流动性简化管理，价格透明 结构性 抗 MEV 靠恒定透明价差，而非算法响应
每个 bin 覆盖固定价格范围（如 0.1%），bin 内价格恒定。
三明治攻击逻辑：攻击者须推爆整个 bin（付出 ≥ bin_step 成本）才能影响价格，但从受害者身上能赚到的滑点 ≤ bin_step。
净收益 ≤ 0，结构上无利可图——无需动态算法，价格离散化本身就是防护。
Uniswap v4 Hooks 完全可编程（开发者自定义） 通用框架，释放创新 框架 Hooks 是工具箱；Likwid 动态费率是针对杠杆借贷风险的专用策略
Likwid 算法动态调整（随市场状态连续变化） 主动风险管理 + 抗 MEV/套利 动态 费率是协议安全系统的一部分，不仅是收益参数

DeFi 中 Anti-MEV 动态费率的主要实现方式

📊

① 基于价格偏离（Likwid 采用）

监控池内价格 vs 外部 TWAP/预言机偏差，超阈值自动上调费率，使套利成本 > 套利利润。

❓ 那价格谁来同步？ 这个机制只阻断「同块内快速操纵」，不阻止「跨块的正常套利」：

慢漂移（Slow Drift）：池内价格因正常交易缓慢偏离市场 → 套利者等下一区块进来 → 此时费率已回落至基础值 → 套利有利可图 → 价格回归 ✓

快速攻击（Rapid Attack）：攻击者在同一区块内大额推价（三明治攻击） → live reserves 瞬间偏离 TWAP → 费率飙升 → 攻击成本 > 收益 → 无利可图 ✗

TWAP 的角色：时间加权平均价格是参照锚，计算过去一段时间的平滑均价，单块操纵无法改变它。动态费率据此判断「本次价格偏离是异常操纵还是正常漂移」，实现精准打击。

📏

② 基于交易规模非线性费率

大额/单向交易按分段函数惩罚性收费，打击大规模套利，避免误伤普通用户。

⏱️

③ 基于时间/区块递增费率

同一池同方向的后续交易在短时间窗口内费率提升，抑制重复套利，给流动性恢复留时间。

🔗

④ 与订单流拍卖集成（OFA）

CowSwap / 1inch Fusion 模式：将 MEV 价值通过拍卖返还给用户，费率机制外包给专业网络。

传统 DEX（公开 mempool） 用户交易广播到公链内存池 → MEV 机器人看到 → 在你前面买入推高价格（front-run）→ 你以更贵的价格成交 → 机器人卖出套利 → 你的损失变成机器人利润

OFA 模式（私密订单 → solver 竞拍） 用户发出「交易意图」（签名报价，不进公开 mempool）→ 专业 solver 竞拍执行权 → 谁能给用户最好成交价谁赢 → 原本被 MEV 机器人拿走的价值，通过竞拍变成给用户的「更优价格」或补贴

按角色快速回顾

读完全文后，根据你的角色定位直接抓取最相关的几条要点，快速对外复述或建模。

🧑‍🌾 如果你是双边 LP（Bilateral LP）
  • 你存入的 ETH+USDT 强制同时承担 swap 做市与杠杆借贷两种角色（见 一），不能选择"我只做 swap"。
  • 收益来自 swap 费（含杠杆开平仓时触发的 swap 费），不直接赚借贷利息（那是单边出借者的专属收益，流入独立的 lend reserves）。
  • 主要风险：IL（市场波动） + 杠杆方坏账（极端行情穿仓损失由 LP 共担），由保险基金兜底（见 六）。
  • 动态费率（见 七）在高波动期自动上调，提升你的实际 APR——但同时意味着市场风险也更高。
  • 对比 Uniswap V3 LP：额外赚取杠杆开平仓带来的 swap 费，但同时承担池深度被杠杆占用后的流动性风险（白皮书注：pair LP 不直接赚借贷利息；是否承担坏账损失目前文档未完全澄清，暂存疑）。
💰 如果你是单边出借者（Single-sided Lender）
  • 只存单一币种，不进入 AMM 曲线，不承担 IL，收益结构最简单：纯借贷利息。
  • 利率随借贷利用率动态变化——杠杆需求旺盛时利率高，冷清时利率低。
  • 极端坏账由保险基金优先吸收（见 六），保险基金耗尽后才会触及本金。
  • 适合稳健型资金：不想做市、不想承担方向性风险，只想拿利率。
  • 对比 Aave 出借：资金被限定服务于本 Vault 的杠杆需求，利用率与 Vault 杠杆活跃度强相关。
📈 如果你是杠杆交易者（Leveraged Trader）
  • 每个 token 仓位需要独立保证金（隔离保证金模型）（见 二），不同仓位风险互不串联。
  • 当 ML < 清算阈值 时强制平仓，协议优先调用 liquidateBurn（无需垫资），失败才走 liquidateCall（见 三 四）。
  • 大仓位开/平仓会直接影响同 Vault 内的 AMM 价格——滑点和反向价格冲击都由你承担。
  • 动态费率在高波动期飙升，开/平仓成本可能远超平静期（见 七），需纳入策略成本。
  • 资金费率持续支付：方向与多数派一致时付费，与少数派一致时收费，是长期持仓的隐性成本/收益。
⚡ 如果你是清算人（Liquidator）
  • liquidateBurn 是首选：只需付 Gas，无需垫资，由协议内部销毁仓位完成清算（见 三）。
  • liquidateCall 适用于外部清算人：需要垫资接盘，对应更高的清算奖励，适合有资金且能与 MEV 争夺的玩家。
  • 当 ML < 1.0（已穿仓）时，没人有经济激励主动清算，此时由保险基金兜底（见 六）——这是 Likwid 与 Aave 最显著的差异。
  • 对比 Aave/GMX：Likwid 的清算市场竞争性更低，参与门槛和奖励结构不同，需单独建模（见 五）。
  • MEV 视角：liquidateBurn 几乎不受三明治攻击影响（无 swap 路径），但 liquidateCall 仍会暴露在抢跑/反向夹击之中。
🔄 如果你是现货交易者（Spot Trader）
  • 动态费率在市场高波动期会显著上升，可能 ≥ 你在 Uniswap 等静态费率 DEX 上的预期成本（见 七）。
  • Vault 内价格由本池 AMM 决定，可能短暂偏离外部 CEX/DEX 市价——套利者最终会把它收敛回来。
  • 同 Vault 内的大额杠杆开/平仓会瞬时推动池内价格，可能导致额外滑点，建议大额单分批。
  • 所有费用最终流向双边 LP + 单边出借者 + 协议金库——你支付的是这些角色的收益来源。
🤖 如果你是套利者（Arbitrageur）
  • 动态费率抑制了同块 MEV（sandwich attack），使攻击成本 > 收益；但跨块套利仍有空间（见 七）。
  • Vault 价格偏离外部市价时有套利机会，但高波动期费率上调会直接压缩利润空间，需动态评估。
  • TWAP 作为费率基准锚点——你的套利收益本质上是"价差收益 − 动态费率成本"，二者同向变化。
  • 对比 Uniswap：Likwid 的动态费率让极端行情下的套利更贵，但也意味着你的竞争对手同样面对相同成本。

八、多空机制使用条件（MemeRush 集成视角）↗ 合约地址

每个 Meme Token 都可以多空吗？ 不是。毕业到 BSC 只代表在 PancakeSwap 有现货流动性，与 Likwid 完全无关。

能否多空取决于两件独立的事：
① 有没有对应的 Likwid Pool——需要人主动部署，不会自动生成
② 这个 Pool 里有没有足够的流动性——需要 LP 和出借人主动注入

目前（2026-05）Likwid BSC 刚上线 26 天，极少数 token 有 Pool，多空覆盖范围非常有限。

Pool 是否随 Token 毕业自动创建？

否。Token 从 Bonding Curve 毕业（Migrate）的过程只会在 PancakeSwap 等 DEX 创建现货流动性池，与 Likwid 协议完全独立。
Likwid Pool 需要额外、显式地调用 LikwidVault.createPool(PoolKey) 才能创建——这是一笔独立的链上交易，可由任何人发起，不需要 Likwid 审批，但必须有人主动去做。

动作触发时机谁触发是否自动
Token 毕业 → PancakeSwap 现货池 Bonding Curve 完成，Migrate 执行 协议自动 ✅ 自动
Likwid Pool 创建 独立，与毕业流程无关 任何人（MemeRush / 项目方 / 用户） ❌ 需手动
LP 注入 pair reserves Pool 创建后，等待 LP 自愿存入 LP 用户 ❌ 需手动

1 Token 已毕业到 BSC 主网 业务前提 从 Bonding Curve 完成 Migrate，在 BSC DEX 有现货流动性
2 对应 Likwid Pool 已创建 无权限门槛，但需要主动部署 调用 LikwidVault.createPool(PoolKey)，参数仅需 {token0, token1, fee, marginFee}；任何人可部署，不随 Token 毕业自动创建
3 Pool fee 高于最低阈值 硬约束（创建时永久确定） 协议有 "low-fee-pool restriction"——PoolKey.fee 低于配置阈值时，LikwidMarginPosition 拒绝为该池开仓，无法事后修改。

如何保证？ 建池时使用 Integration Manual 规定的标准档位即可：fee = 3000（0.3%） 或 fee = 5000（0.5%），均高于阈值，margin 功能永久可用。

文档来源： Margin Position Mechanics > Fee Structure（白皮书原文："the current manager rejects margin creation on pools whose base LP fee is below the configured threshold for margin usage"）。Exact 阈值数值未在文档中公布，但使用标准档位无风险。
4 Lend reserves ≥ 借入金额 硬约束（按单笔校验） 开仓时合约校验 lend_reserves ≥ borrow_amount；不满足则 revert，仓开不了。利用率高 → 借款利率飙升，形成市场软约束。
不随 Token 毕业自动满足——需要出借人主动调用 LikwidLendPosition.addSingleLiquidity() 注入资金
5 Pair reserves（AMM 深度） 软约束（影响质量） 无金额下限——协议层不拒绝，但越薄滑点越大。Pair reserves 耗尽时启用降级机制：改用 lA 代币（利息计息版）替代继续交易，不直接失败但用户体验变差。
不随 Token 毕业自动满足——需要 LP 主动调用 LikwidPairPosition.addLiquidity() 双边注入

"足够"的实际含义

流动性类型硬下限实用参考标准
Lend reserves 必须 ≥ 单笔最大借入量
（否则开仓失败）		 按用户可开最大仓位估算：
支持 5× 杠杆、单仓 $1000 USDT → lend reserves 需 ≥ $4000（5×−1=4倍借入）
建议初期注入 $10K+ 以支持基础交易
Pair reserves 无协议下限（空池不 revert） 按可接受滑点反推：
$1000 仓位、5% 滑点容忍 → pair reserves 需 ≥ ~$20K
池子越深，价格越难被操纵，套利者越能维持池内/外价格一致

实操 FAQ

问题答案
fee 设置错了（太低）怎么知道？ 建池前可通过 LikwidMarginPosition 合约（0x6bec0c1dc4898484b7F094566ddf8bC82ED7Abe8）读取协议配置常量来验证，或直接采用 Integration Manual 标准档 fee = 3000 规避风险。fee 一旦写入 PoolKey 就是 poolId 的一部分，建错了只能重建新 Pool（旧 Pool 永久不支持 margin）。
毕业后 lend reserves 和 pair reserves 能自动满足吗？ 不能。 毕业流程只在 PancakeSwap 创建现货池，与 Likwid 完全独立。Likwid Pool 的两类流动性均需主动注入：
① Pair reserves：LP 双边存入（token + USDT），通过 LikwidPairPosition.addLiquidity()
② Lend reserves：出借人单边存入 USDT，通过 LikwidLendPosition.addSingleLiquidity()
两者都需要真实资金承诺，这是 Phase 2 的核心商业问题（谁来出钱、用什么激励吸引流动性），而不是技术问题。

扩大覆盖范围的真正瓶颈：流动性引导

误解实际情况
"需要 Likwid BD 同意才能新增 token" ❌ 错误。Pool 创建无许可，MemeRush 可以自行为任何毕业 token 建池
"建了 Pool 就能马上交易" ❌ 错误。空池子无法开杠杆仓，必须先有 LP 注入 pair reserves 和 lend reserves
"覆盖范围扩大 = 技术工作" ✅ 部分正确。技术上 1 笔交易创建池，但流动性引导是商业/激励问题：谁来当初始 LP？补贴多少？
Hackathon 实践建议 Demo 阶段：为 1-2 个 BSC 毕业 meme token 手动建池 + 注入少量测试资金（可用 BSC testnet 或正式网极小金额）。 Phase 2：与 Likwid 合作以 LP 激励计划快速引导多 token 覆盖，不依赖 Likwid "批准"，而是共同做大流动性饼。
